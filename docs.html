Спецификация интеграции Core API

Настоящий документ содержит исчерпывающее техническое описание интерфейсов взаимодействия распределенной сети ShadowsLTD Core. Платформа спроектирована для обеспечения атомарных транзакций обмена метаданными между изолированными транспортными узлами (Edge Nodes) в условиях асимметричных сетевых задержек.

1. Протокол криптографической верификации

Все входящие вызовы к магистральным шлюзам верхнего уровня требуют обязательной пре-авторизации пакетов в рамках нулевого транспортного слоя (L0). Сессионное рукопожатие осуществляется с использованием гибридного шифрования на базе эфемерных ключей Диффи-Хеллмана на эллиптических кривых (ECDHE-X25519) со сквозным контролем целостности HMAC-SHA256.

// Пример генерации токена пограничной авторизации узла ShadowsLTD { "auth_handshake": { "node_uuid": "f81d4fae-7dec-11d0-a765-00a0c91e6bf6", "timestamp_epoch": 1774944000, "ephemeral_public_key": "0x7a9d4f21b3e8c...aefc88" } }

2. Параметры авторизационных заголовков

Передача служебных метаданных должна строго соответствовать RFC-инструкциям кластера. Использование кастомных прокси-заголовков приводит к немедленной инвалидации сессионного конвейера балансировщика трафика.

Заголовок Тип Описание
X-Core-Node-ID String (Hex) Уникальный криптографический идентификатор верифицированного узла ShadowsLTD.
X-Routing-Signature String (Base64) Подпись тела запроса, вычисленная с использованием приватного ключа сессии.

3. Метод /v4/node/sync

Служебный эндпоинт, предназначенный для периодической низкоуровневой синхронизации топологии распределенной сети. Запросы принимаются исключительно от доверенных IP-адресов, прошедших верификацию в локальном файрволе.

4. Метод /v4/session/init

Инициализирует новый защищенный контекст для последующего пропуска инкапсулированных пакетов данных. Возвращает уникальный сессионный токен со временем жизни (TTL) 3600 секунд.

5. Децентрализованная топология Mesh-сетей

Сетевая модель ShadowsLTD базируется на гибридной mesh-топологии. Каждый подключенный узел (Edge Node) автоматически выполняет пиринговое сканирование соседних сегментов для построения оптимальной карты маршрутизации (Routing Matrix). Обмен служебными пакетами верификации путей происходит по протоколу UDP, что минимизирует накладные расходы на поддержание сессий.

При обнаружении деградации транзитного узла система автоматически перестраивает виртуальные туннели через альтернативные аплинки без разрыва активных клиентских сокетов.

6. Метод /v4/tunnel/open

Высокоуровневый системный вызов для динамического резервирования виртуальных портов инкапсуляции. Используется исключительно внутренними демонами ядра для оркестрации зашифрованных потоков.

// Request Payload (Инициализация мультиплексированного канала) { "tunnel_config": { "protocol_family": "UDP_ENCAP", "target_port_range": "40000-50000", "congestion_control": "BBRv3", "mtu_discovery": true } }

7. Регламент аварийного переключения (Failover State)

В случае фиксации критических перегрузок на магистральных каналах (утилизация пула очередей L3/L4 > 92%), пограничные шлюзы автоматически переводят затронутые кластеры в режим **DEGRADED**.

В данном состоянии активируются алгоритмы агрессивного сжатия заголовков и включается динамическое ограничение пропускной способности (Rate Limiting) для неавторизованных внешних источников, что позволяет сохранить общую связность распределенной инфраструктуры.
